HTML/CSS/JavaScript — พื้นฐาน Frontend

สัปดาห์ที่ 1 · 1-2 ชม./วัน
▼

📖 Concept ที่ต้องรู้

Conceptคืออะไรทำไม PM ต้องรู้
HTMLโครงสร้างหน้าเว็บ (ปุ่ม, ช่องกรอก, ตาราง)รู้ว่าหน้าจอนี้มีอะไรบ้าง
CSSตกแต่งหน้าเว็บ (สี, ขนาด, ตำแหน่ง)สั่ง Designer/Dev ได้ชัดเจน
JavaScriptทำให้หน้าเว็บ "ทำงานได้" (กดปุ่ม → เกิดอะไร)เข้าใจ Flow การทำงาน
DOMตัวแทนของหน้าจอในโค้ดรู้ว่า Dev เปลี่ยนหน้าจอยังไง
Responsiveหน้าเว็บปรับตามขนาดจอทดสอบ Mobile/Tablet/Desktop ได้
💡 ไม่ต้องเขียนเป็น — แค่ "อ่านออก" ว่าโค้ดนี้ทำอะไร

📝 สิ่งที่ต้องทำ

  1. ดูวิดีโอ "HTML & CSS in 100 Seconds" (YouTube) — 5 นาที
  2. ดูวิดีโอ "JavaScript in 100 Seconds" — 5 นาที
  3. เปิดโค้ดหน้าเว็บใดๆ (คลิกขวา → View Source) — ลองอ่านดู
  4. ส่งโค้ดมาให้ผมอธิบาย ว่าแต่ละบรรทัดทำอะไร
  • ดูวิดีโอ HTML/CSS/JS (15 นาที)
  • เปิด View Source หน้าเว็บ — ลองอ่าน
  • เข้าใจว่า HTML = โครงสร้าง, CSS = ตกแต่ง, JS = ทำงาน

Framework Frontend — React, Vue, Angular

สัปดาห์ที่ 1-2 · 1 ชม./วัน
▼

📖 Concept ที่ต้องรู้

Frameworkใครใช้จุดเด่น
ReactFacebook, Netflix, Airbnbยืดหยุ่น, Community ใหญ่
VueAlibaba, Xiaomiเรียนรู้ง่าย, เบา
AngularGoogle, Microsoftครบวงจร, เหมาะกับ Enterprise
FlutterGoogle, BMWเขียนครั้งเดียว ได้ทั้ง iOS+Android

🔑 Concept สำคัญ

  • Component = ชิ้นส่วน UI ที่ใช้ซ้ำได้ (เช่น ปุ่ม, การ์ด, Navbar)
  • State = ข้อมูลที่เปลี่ยนแปลงได้ (เช่น จำนวนในตะกร้า)
  • Props = ข้อมูลที่ส่งระหว่าง Component
  • Routing = การเปลี่ยนหน้า
💡 สำหรับ PM: รู้ว่า App ใช้ Framework อะไร → รู้ว่า Developer หาคนแทนง่ายไหม, Community มีไหม
ฉันเป็น Product Manager App ของฉันใช้ React ช่วยอธิบายให้ฉันเข้าใจว่า: 1. React ทำงานยังไง (แบบง่ายๆ) 2. Component, State, Props คืออะไร 3. ทำไมทีมถึงเลือก React แทน Vue/Angular 4. ฉันต้องรู้อะไรบ้างเพื่อตรวจงาน Developer
  • รู้ว่า App ของตัวเองใช้ Framework อะไร
เข้าใจ Component, State, Props, Routing
  • บอกได้ว่าทำไมทีมเลือก Framework นี้

    API — ตัวเชื่อม Frontend ↔ Backend

    สัปดาห์ที่ 2 · 1-2 ชม./วัน
    ▼

    📖 Concept ที่ต้องรู้

    Conceptคืออะไรตัวอย่าง
    APIตัวเชื่อมระหว่าง Frontend กับ Backendหน้า Login → ส่ง username/password → API ตรวจสอบ → ตอบกลับมา
    REST APIรูปแบบ API ที่นิยมที่สุดGET /users, POST /login, PUT /users/1, DELETE /users/1
    HTTP MethodsGET (อ่าน), POST (สร้าง), PUT (แก้ไข), DELETE (ลบ)CRUD Operations
    Request/Responseส่งอะไรไป → ได้อะไรกลับมาส่ง {username, password} → ได้ {token, user}
    Status Code200 = สำเร็จ, 400 = ข้อมูลผิด, 401 = ไม่มีสิทธิ์, 500 = Server พังรู้ได้ว่า Error เกิดจากอะไร

    🔑 ทำไม PM ต้องรู้ API?

    • เขียน API Spec ให้ Developer ได้
    • รู้ว่า Frontend ต้องเรียก API อะไรบ้าง
    • อ่าน Error Log ได้ → บอก Dev ถูกว่าปัญหาอยู่ที่ไหน
    • ทดสอบ API เองได้ (ใช้ Postman)
    ช่วยออกแบบ API Spec สำหรับ App "[ชื่อ App]" โดยรวมถึง: - ทุก Endpoint ที่ต้องมี - HTTP Method (GET/POST/PUT/DELETE) - Request Body (ส่งอะไรไป) - Response (ได้อะไรกลับมา) - Status Code (200, 400, 401, 500) แสดงเป็นตาราง
    • เข้าใจ GET, POST, PUT, DELETE
    • เข้าใจ Status Code (200, 400, 401, 500)
    • อ่าน API Spec ออก
    • เขียน API Spec เบื้องต้นได้ (ด้วย AI)

    Server, Logic, Authentication

    สัปดาห์ที่ 3 · 1 ชม./วัน
    ▼

    📖 Concept ที่ต้องรู้

    Conceptคืออะไรทำไม PM ต้องรู้
    Serverคอมพิวเตอร์ที่รัน Backendรู้ว่าจะต้อง Deploy ที่ไหน
    Business Logicกฎการทำงาน (เช่น คำนวณราคา, ตรวจสอบสิทธิ์)เขียน Requirement ได้ชัดเจน
    Authenticationยืนยันตัวตน (Login)รู้ว่าระบบ Login ปลอดภัยไหม
    Authorizationตรวจสอบสิทธิ์ (Admin vs User)ออกแบบ Role/Permission ได้
    JWT/Tokenบัตรผ่านดิจิทัล หลัง Loginเข้าใจว่า Session ทำงานยังไง
    Middlewareตัวกรองก่อนเข้า Backend (เช่น ตรวจสอบ Token)เข้าใจ Flow การทำงาน
    💡 สำหรับ PM: ไม่ต้องเขียน Backend เป็น แต่ต้องรู้ว่า "Backend ต้องทำอะไรบ้าง" และ "ทดสอบยังไงว่าทำงานถูกต้อง"
    • เข้าใจ Server, Business Logic, AuthN/AuthZ
    • เข้าใจ JWT/Token ทำงานยังไง
    • ออกแบบ Role/Permission เบื้องต้นได้

    📊

    Relational Database — ตาราง & ความสัมพันธ์

    สัปดาห์ที่ 3 · 1-2 ชม./วัน
    ▼

    📖 Concept ที่ต้องรู้

    Conceptคืออะไรตัวอย่าง
    Tableตารางเก็บข้อมูลusers, orders, products
    Column/Fieldคอลัมน์ในตารางid, name, email, created_at
    Row/Recordแถวข้อมูล1 แถว = 1 ผู้ใช้
    Primary Keyคอลัมน์ที่ระบุแถวไม่ซ้ำid (auto increment)
    Foreign Keyคอลัมน์ที่เชื่อมตารางอื่นorders.user_id → เชื่อม users.id
    Relationship1:1, 1:N, N:M1 ผู้ใช้ มี หลายออเดอร์ (1:N)
    ER Diagramแผนภาพความสัมพันธ์เห็นภาพรวม Database ทั้งหมด
    Indexทำให้ค้นหาเร็วขึ้นIndex on email → ค้นหาด้วย email เร็ว
    จาก Requirement Document ของ App "[ชื่อ App]" ช่วยออกแบบ Database Schema ให้ โดยรวมถึง: - ตารางทั้งหมดที่ต้องมี - คอลัมน์ในแต่ละตาราง (ชื่อ, ประเภท, เป็น PK/FK ไหม) - ความสัมพันธ์ระหว่างตาราง - แสดงเป็น ER Diagram (Text-based)
    • เข้าใจ Table, Column, Row, PK, FK
    • เข้าใจ 1:1, 1:N, N:M Relationship
    • อ่าน ER Diagram ออก
    • ออกแบบ Database Schema เบื้องต้นได้ (ด้วย AI)

    SQL เบื้องต้น + NoSQL

    สัปดาห์ที่ 4 · 1 ชม./วัน
    ▼

    📖 SQL ที่ PM ต้องรู้

    คำสั่งทำอะไรตัวอย่าง
    SELECTอ่านข้อมูลSELECT * FROM users WHERE age > 18;
    INSERTเพิ่มข้อมูลINSERT INTO users (name, email) VALUES ('Nan', 'nan@pkg.com');
    UPDATEแก้ไขข้อมูลUPDATE users SET email = 'new@pkg.com' WHERE id = 1;
    DELETEลบข้อมูลDELETE FROM users WHERE id = 1;
    JOINเชื่อมตารางSELECT * FROM orders JOIN users ON orders.user_id = users.id;

    NoSQL vs SQL

    SQL (Relational)NoSQL
    ตัวอย่างPostgreSQL, MySQL, SQLiteMongoDB, Firebase, Supabase
    โครงสร้างตาราง (Table) — ข้อมูลมีโครงสร้างชัดเจนDocument/Key-Value — ยืดหยุ่นกว่า
    เหมาะกับข้อมูลที่มีความสัมพันธ์ชัดเจน (User-Order-Product)ข้อมูลที่โครงสร้างไม่คงที่
    PM ต้องรู้ออกแบบ ER Diagram → ให้ Dev สร้างรู้ว่าเมื่อไหร่ควรใช้ NoSQL
    💡 สำหรับ PM: ไม่ต้องเขียน SQL เองเป็น แต่ต้อง "อ่านออก" ว่า Query นี้ทำอะไร และรู้ว่า Dev ดึงข้อมูลมาถูกต้องไหม
    • อ่าน SQL SELECT, INSERT, UPDATE, DELETE ออก
    • เข้าใจ JOIN ว่าเชื่อมตารางยังไง
    • เข้าใจความแตกต่าง SQL vs NoSQL
    • เขียน SQL ง่ายๆ ได้ (ด้วย AI)

    📖 Architecture Patterns ที่ต้องรู้

    Patternคืออะไรเหมาะกับ
    Monolithicทุกอย่างอยู่ใน App เดียวApp เล็ก-กลาง, เริ่มต้นเร็ว

    🔑 Tech Stack Decision

    PM ต้องรู้ว่าแต่ละ Layer ใช้อะไร และทำไม:

    Layerตัวเลือกข้อดีข้อเสีย
    FrontendReact / Vue / FlutterReact = Community ใหญ่React = เรียนรู้ยากกว่า
    BackendNode.js / Python / GoNode.js = ใช้ JS เดียวกันPython = ช้ากว่า
    DatabasePostgreSQL / MongoDBPostgreSQL = มาตรฐานMongoDB = Query ยาก
    CloudAWS / GCP / VercelVercel = ง่ายสุดAWS = ยืดหยุ่นกว่า
    จาก Requirement Document ของ App "[ชื่อ App]" ช่วยแนะนำ System Architecture ที่เหมาะสม โดยรวมถึง: - Architecture Pattern ที่แนะนำ + เหตุผล - Tech Stack แต่ละ Layer (Frontend, Backend, Database, Cloud) - Diagram แสดงการเชื่อมต่อกัน - ข้อดี-ข้อเสีย ของแต่ละตัวเลือก - ประมาณค่าใช้จ่ายต่อเดือน
    • เข้าใจ Monolithic, Client-Server, Microservices
    • เข้าใจ Tech Stack แต่ละ Layer
    • อ่าน Architecture Diagram ออก
    • ตัดสินใจเลือก Tech Stack ได้ (ร่วมกับ Dev)

    🎭 Design / UI/UX — ออกแบบประสบการณ์ผู้ใช้

    🎨

    UI/UX Fundamentals

    สัปดาห์ที่ 5 · 1-2 ชม./วัน
    ▼

    📖 Concept ที่ต้องรู้

    Conceptคืออะไรทำไม PM ต้องรู้
    UI (User Interface)หน้าตาของ App (สี, ฟอนต์, ปุ่ม)ตรวจงาน Designer ได้
    UX (User Experience)ความรู้สึกเมื่อใช้ App (ง่ายไหม, สับสนไหม)ตัดสินใจเรื่อง Flow ได้
    Wireframeโครงร่างหน้าจอ (ไม่มีสี/รูป)คุยกับ Dev ก่อนออกแบบจริง
    Mockupหน้าจอที่มีสี/รูป เหมือนจริงเห็นหน้าตาจริงก่อนสร้าง
    Prototypeหน้าจอที่กดได้ (เหมือน App จริง)ทดสอบก่อนเขียนโค้ด
    Design Systemชุดกฎการออกแบบ (สี, ฟอนต์, Component)รักษาความสม่ำเสมอ
    User Flowเส้นทางที่ผู้ใช้เดินใน Appรู้ว่าต้องสร้างกี่หน้าจอ
    Accessibilityทำให้ทุกคนใช้ได้ (รวมถึงผู้พิการ)กฎหมาย + คุณธรรม

    🔑 กฎ UI/UX พื้นฐาน

    • Less is More — หน้าจอที่เรียบง่าย = ใช้ง่าย
    • Consistency — ปุ่มเดียวกัน หน้าตาเดียวกัน ทุกหน้า
    • Feedback — ทุก Action ต้องมีตอบกลับ (Loading, Success, Error)
    • Hierarchy — สิ่งสำคัญต้องเด่นที่สุด
    • 3-Click Rule — ผู้ใช้ควรหาสิ่งที่ต้องการได้ใน 3 คลิก
    💡 สำหรับ PM: ไม่ต้องออกแบบเองเป็น แต่ต้อง "ดูออก" ว่า Design นี้ดีไหม และบอก Designer/Dev ได้ว่าต้องแก้ตรงไหน
    • เข้าใจ UI vs UX, Wireframe vs Mockup vs Prototype
    • เข้าใจกฎ UI/UX พื้นฐาน 5 ข้อ
    • ตรวจงาน Designer ได้ (ดูออกว่าดี/ไม่ดี)
    • สร้าง Wireframe เบื้องต้นได้ (ด้วย AI/Figma)

    🔒 Security — ความปลอดภัย

    สัปดาห์ที่ 6-7 · รู้จักช่องโหว่ มาตรการป้องกัน

    🛡️

    Security Basics — ช่องโหว่ที่พบบ่อย

    สัปดาห์ที่ 6 · 1-2 ชม./วัน
    ▼

    📖 ช่องโหว่ OWASP Top 10 ที่ PM ต้องรู้

    #ช่องโหว่คืออะไรป้องกันยังไง
    1SQL Injectionแฮกเกอร์ใส่ SQL ผ่านฟอร์ม → เข้า Database ได้ใช้ Parameterized Query
    2XSSแฮกเกอร์ใส่ JavaScript → ขโมยข้อมูลผู้ใช้Escape Input, Content Security Policy
    3Broken Authenticationระบบ Login ไม่ปลอดภัย → แฮกเข้าสู่ระบบใช้ JWT, Multi-Factor Auth, Rate Limiting
    4Broken Access ControlUser เข้าถึงข้อมูลของคนอื่นได้ตรวจสอบสิทธิ์ทุก Request
    5Sensitive Data Exposureข้อมูลสำคัญไม่เข้ารหัส → ถูกดักจับHTTPS, Encrypt Password (bcrypt)

    🔑 Security Checklist สำหรับ PM

    • ✅ ใช้ HTTPS ทุกหน้า
    • ✅ Password เข้ารหัส (bcrypt, ไม่เก็บ Plain Text)
    • ✅ มี Rate Limiting (ป้องกัน Brute Force)
    • ✅ Input Validation ทุกฟอร์ม
    • ✅ Session Timeout (อัตโนมัติ Logout)
    • ✅ Backup Database สม่ำเสมอ
    • ✅ Log ทุก Action สำคัญ
    • ✅ PDPA Compliance (ขอ Consent, ลบข้อมูลได้)
    จาก App "[ชื่อ App]" ช่วยวิเคราะห์ Security Risk โดยรวมถึง: 1. ช่องโหว่ที่อาจมี (ตาม OWASP Top 10) 2. มาตรการป้องกันที่แนะนำ 3. Security Checklist ที่ต้องตรวจสอบก่อน Deploy 4. PDPA Requirements ที่ต้องทำ
    • เข้าใจ OWASP Top 10 (5 อันดับแรก)
    • เข้าใจ HTTPS, JWT, bcrypt
    • รู้ว่าต้องถาม Dev ว่า "ป้องกัน SQL Injection ยังไง?"
    • เข้าใจ PDPA Requirements พื้นฐาน

    Authentication & Authorization

    สัปดาห์ที่ 7 · 1 ชม./วัน
    ▼

    📖 Concept ที่ต้องรู้

    Conceptคืออะไรตัวอย่าง
    Authentication (AuthN)ยืนยันตัวตน = "คุณคือใคร?"Login ด้วย Username/Password
    Authorization (AuthZ)ตรวจสอบสิทธิ์ = "คุณทำอะไรได้บ้าง?"Admin ลบข้อมูลได้, User ดูได้อย่างเดียว
    Sessionสถานะการ Loginหลังจาก Login → Server จำว่าคุณ Login แล้ว
    Token (JWT)บัตรผ่านดิจิทัลLogin สำเร็จ → ได้ Token → ใช้ Token เข้าหน้าอื่น
    OAuth / SSOLogin ด้วยบัญชีอื่น (Google, Facebook)"Login with Google"

    สรุป — สิ่งที่ PM ต้องรู้ในแต่ละด้าน

    ไม่ต้องเขียนเองเป็น — แต่ต้องเข้าใจ ตัดสินใจได้ ตรวจงานได้

    ด้านต้องรู้ไม่ต้องรู้ถาม Dev ว่าอะไร
    🎨 Frontend HTML/CSS/JS ทำงานยังไง, Framework ที่ใช้, Component, State เขียนโค้ด CSS สวยๆ, จำ Syntax "ใช้ Framework อะไร? ทำไม? Responsive ไหม?"
    ⚙️ Backend API, HTTP Methods, Status Code, AuthN/AuthZ, Business Logic เขียน Backend เอง, Setup Server "API มีกี่ Endpoint? ป้องกัน SQL Injection ยังไง?"
    🗄️ Database Table, Column, PK, FK, Relationship, SQL เบื้องต้น, ER Diagram เขียน Query ซับซ้อน, Optimize Database "Database Schema เป็นยังไง? มี Index ตรงไหน?"
    🏗️ Architecture Pattern (Monolithic, Microservices), Tech Stack แต่ละ Layer ออกแบบ Architecture เอง, Setup Infrastructure "ใช้ Architecture แบบไหน? ทำไม? Scalable ไหม?"
    🎭 Design/UI/UX UI vs UX, Wireframe, User Flow, Design System, กฎพื้นฐาน ออกแบบเองใน Figma, ทำ Prototype สวยๆ "User Flow เป็นยังไง? ผ่าน User Testing แล้วหรือยัง?"
    🔒 Security OWASP Top 10, HTTPS, JWT, bcrypt, PDPA, RBAC เขียนโค้ดป้องกันเอง, Penetration Testing "ป้องกัน XSS/SQL Injection ยังไง? Password เข้ารหัสไหม?"
