🎯 UX Principles — หลักการออกแบบ

ทุกการตัดสินใจมีเหตุผล มาจากปัญหาจริงของผู้ใช้ ไม่ใช่ความสวยงาม

1️⃣

ผู้ใช้ไม่ต้องคิด — เห็นแล้วเข้าใจทันที

💡 ทำไม?

ผู้ใช้ PP5 มีตั้งแต่สมาชิกใหม่จนถึงผู้บริหาร — ถ้าต้อง "คิด" ว่าต้องกดอะไร ระบบล้มเหลวทันที Dashboard ต้องตอบคำถาม 3 ข้อใน 3 วินาที: (1) ฉันอยู่ไหน (2) ต้องทำอะไร (3) สถานะเป็นยังไง

✅ ดีไซน์ที่เลือก

ใช้ "Progressive Disclosure" — แสดงข้อมูลสำคัญก่อน (KPI 4 ตัว) → กดดูรายละเอียดได้ทีหลัง ไม่ยัดทุกอย่างในหน้าเดียว

2️⃣

งานที่เคยทำ 10 นาที → ทำให้เสร็จใน 30 วินาที

💡 ทำไม?

ADMIN PAD ต้องอนุมัติงาน ตรวจสอบสถานะ ดูรายงาน — ถ้าทุกงานต้องกด 5-6 คลิก ADMIN จะเลิกใช้ระบบ และกลับไปใช้ Excel

✅ ดีไซน์ที่เลือก

1 คลิก = ดูรายละเอียด | 2 คลิก = อนุมัติ/ปฏิเสธ | 3 คลิก = ส่งรายงาน ทุกงานสำคัญอยู่ในหน้าแรก ไม่ต้องค้นหา

3️⃣

แยกมุมมอง ADMIN / สมาชิก — ไม่ปนกัน

💡 ทำไม?

ADMIN ต้องการ "จัดการ + อนุมัติ + ดูภาพรวม" | สมาชิกต้องการ "ดูของตัวเอง + สมัคร + ส่งงาน" — ถ้ารวมเมนูเดียวกัน ADMIN จะเห็นเมนูที่ไม่ใช้ 50% สมาชิกจะเห็นเมนูที่กดไม่ได้ 70%

✅ ดีไซน์ที่เลือก

ใช้ Role-Based Navigation — เมนูต่างกันตาม Role แต่ใช้ Layout เดียวกัน (เรียนรู้ครั้งเดียว ใช้ได้ทั้งสองบทบาท)

4️⃣

สี = ความหมาย ไม่ใช่ตกแต่ง

💡 ทำไม?

ผู้ใช้สแกนหน้าจอก่อนอ่าน — สีต้องสื่อสถานะทันที: เขียว = ผ่าน, เหลือง = รอ, แดง = ปัญหา, น้ำเงิน = ข้อมูล ถ้าใช้สีมั่ว ผู้ใช้จะสับสน

❌ อย่าทำ

ใช้สีตามความสวยงาม, ใช้ Gradient ทุกรายการ, ใช้สีแดงสำหรับหัวข้อ (ผู้ใช้จะคิดว่าเป็นปัญหา)

✅ ดีไซน์ที่เลือก

สีหลัก: #0f172a (Neutral) | #2563eb (Action) | #059669 (Success) | #d97706 (Warning) | #dc2626 (Danger) — ทุกสีมีความหมายชัดเจน

5️⃣

Mobile-First แต่ Desktop-Rich

💡 ทำไม?

สมาชิก 60% ดูผ่านมือถือ (ดูผลตัวเอง, สมัครอบรม) | ADMIN 80% ใช้ Desktop (จัดการ, อนุมัติ, รายงาน) — ออกแบบให้มือถือใช้ง่ายก่อน แล้วค่อยเพิ่มฟีเจอร์สำหรับ Desktop

✅ ดีไซน์ที่เลือก

Mobile: Top Nav + Stacked Cards (กดง่าย) | Desktop: เพิ่ม Sidebar สำหรับ ADMIN (จัดการได้เร็วขึ้น)

👥 User Personas — รู้จักผู้ใช้

ออกแบบตามความต้องการจริง ไม่ใช่สมมติ

🔧

ADMIN PAD

ผู้จัดการระบบ HRD / Supervisor
  • เป้าหมาย: ดูภาพรวม, อนุมัติงาน, ติดตามสถานะสมาชิก
  • ความถี่: ใช้ทุกวัน (10-30 นาที/วัน)
  • อุปกรณ์: Desktop 80%, Tablet 20%
  • ปัญหา: ต้องเปิดหลายหน้าเพื่ออนุมัติงาน, หาข้อมูลสมาชิกไม่เจอ
  • Success Metric: อนุมัติงานได้ภายใน 2 คลิก
👤

สมาชิก / BMC

พนักงานทุกระดับ
  • เป้าหมาย: ดูผลตัวเอง, สมัครอบรม, ส่งงาน, ดูความคืบหน้า
  • ความถี่: ใช้สัปดาห์ละ 1-2 ครั้ง (2-5 นาที/ครั้ง)
  • อุปกรณ์: Mobile 60%, Desktop 40%
  • ปัญหา: ไม่รู้ว่าต้องอบรมอะไร, หาผลสอบไม่เจอ
  • Success Metric: ดูผลตัวเองได้ภายใน 1 คลิก

ADMIN PAD Navigation

💡 ทำไมจัดแบบนี้?

ADMIN ต้องการ "จัดการ → ติดตาม → รายงาน → ตั้งค่า" — เรียงตามลำดับการใช้งานจริง (งานประจำ → งานวิเคราะห์ → งานตั้งค่า)

🏠 หน้าแรก→ 👥 สมาชิก→ 📚 หลักสูตร→ ✅ อนุมัติงาน→ 📊 รายงาน→ 🔄 วงจรการเรียนรู้→ ⚙️ ตั้งค่า
เมนูงานหลักความถี่เหตุผล
🏠 หน้าแรกเห็นงานค้าง, KPI ด่วนทุกครั้งเป็น Landing Page — ไม่ต้องเลือกเมนู
👥 สมาชิกดู/แก้ไข/เพิ่มสมาชิกทุกวันงานหลักของ ADMIN
📚 หลักสูตรสร้าง/แก้ไข/กำหนดทุนสัปดาห์ละครั้งจัดการเนื้อหา
✅ อนุมัติงานอนุมัติ/ปฏิเสธ/มอบหมายทุกวันแยกออกมา — งานสำคัญที่สุด
📊 รายงานKPI, Progress, Exportสัปดาห์ละครั้งงานวิเคราะห์
🔄 วงจรการเรียนรู้ดู Flow การพัฒนาเดือนละครั้งงานวางแผน
⚙️ ตั้งค่าเกณฑ์, สิทธิ์, Backupเดือนละครั้งงานตั้งค่า — ซ่อนไว้
👤

สมาชิก/BMC Navigation

💡 ทำไมจัดแบบนี้?

สมาชิกต้องการ "ดูของฉัน → สมัคร → ส่งงาน → ดูผล" — เรียงตามลำดับที่ผู้ใช้ทำจริง (ดู → ทำ → ดูผล)

🏠 หน้าแรก→ 📚 หลักสูตรของฉัน→ 📝 สมัครอบรม→ 📤 ส่งงาน→ 📈 Progress→ 👤 โปรไฟล์
เมนูงานหลักความถี่เหตุผล
🏠 หน้าแรกเห็นหลักสูตรถัดไป, งานค้างทุกครั้งเป็น Landing Page
📚 หลักสูตรของฉันดูผล, ดูหลักสูตรที่ผ่าน/รอทุกครั้งหน้าที่ใช้บ่อยที่สุด
📝 สมัครอบรมเลือก + สมัครเดือนละครั้งแยกออกมา — งานสำคัญ
📤 ส่งงานอัปโหลด Role Play, งานตามกำหนดงานที่ต้องทำ
📈 Progressดู Timeline, ทุน, เกรดเดือนละครั้งดูการเติบโต
👤 โปรไฟล์ข้อมูลส่วนตัว, เปลี่ยนรหัสเดือนละครั้งจัดการตัวเอง

📐 Layout Decision — ทำไมเลือกแบบนี้

เปรียบเทียบ 3 แบบ → เลือก 1 แบบที่ดีที่สุด

🏆

✅ เลือก: Top Navigation + Content Area (Layout 2)

✅ เหตุผลที่เลือก

1. เหมาะกับทั้ง 2 บทบาท — เมนูไม่เยอะ (6-7 รายการ) ทั้ง ADMIN และสมาชิกใช้เมนูจำนวนใกล้เคียงกัน
2. มือถือได้ดี — Top Nav ซ่อนใน Hamburger Menu ได้ง่าย Sidebar ทำยาก
3. พื้นที่เนื้อหาเยอะ — Dashboard, ตาราง, กราฟ ต้องการพื้นที่กว้าง
4. ขยายได้ — เพิ่มเมนูได้โดยไม่รก (Dropdown ได้)
5. ผู้ใช้คุ้นเคย — ทุกเว็บสมัยใหม่ใช้ Top Nav — ไม่ต้องเรียนรู้ใหม่

❌ ทำไมไม่เลือก Sidebar (Layout 1)

Sidebar เหมาะกับระบบที่มีเมนู 10+ รายการ — PP5 มีแค่ 6-7 เมนู Sidebar จะเปลืองพื้นที่ 240px โดยไม่จำเป็น และบนมือถือต้องซ่อน → ใช้งานยากกว่า

❌ ทำไมไม่เลือก Tabs (Layout 3)

Tabs เหมาะกับหน้าเดียวที่สลับ View — PP5 มีหลายหน้า/หลาย Section Tabs จะจำกัดจำนวนเมนู (ไม่เกิน 5-6) และดูไม่เป็นระบบ

📊 Dashboard Design — ทำไมออกแบบแบบนี้

Dashboard ที่ดี = เห็นทุกอย่างที่สำคัญ ในหน้าเดียว โดยไม่ต้องเลื่อน

🎯

หลักการออกแบบ Dashboard

1. เห็นใน 3 วินาที

KPI 4 ตัวบนสุด — ตัวใหญ่ สีชัดเจน ไม่ต้องอ่านคำอธิบายก็เข้าใจ

2. ทำงานได้ทันที

งานค้างอนุมัติอยู่ใน Dashboard — ไม่ต้องกดไปหน้าอื่น อนุมัติได้เลยจากหน้าแรก

3. ดูแนวโน้มได้

Progress Bar แต่ละทุน — เห็นว่า BU ไหนต้องติดตาม สีเปลี่ยนตามสถานะ

📊

Dashboard Layout — ADMIN PAD

ส่วนเนื้อหาเหตุผลลำดับ
1. KPI Barสมาชิก, ผ่านเกณฑ์, หลักสูตร, ชม.อบรมเห็นภาพรวมทันทีสูงสุด
2. งานค้างอนุมัติรายการรออนุมัติ + ปุ่มอนุมัติ/ปฏิเสธงานสำคัญที่สุดของ ADMINสูงสุด
3. KPI/TA/ProgressProgress Bar แต่ละทุน (องค์กร/มนุษย์/สารสนเทศ)เห็นแนวโน้มแต่ละ BUสูง
4. ตารางสมาชิก5 ลำดับล่าสุด + สถานะ + ปุ่มดูรายละเอียดติดตามรายบุคคลปานกลาง
5. รายงานด่วนExport PDF/Excelส่งผู้บริหารต่ำ
👤

Dashboard Layout — สมาชิก

ส่วนเนื้อหาเหตุผลลำดับ
1. Progress Cardsหน่วยกิต, GPA, ชม.อบรม, ผ่านที่ 3เห็นสถานะตัวเองทันทีสูงสุด
2. หลักสูตรถัดไปวันที่, วิทยากร, สถานที่รู้ว่าต้องทำอะไรสูงสุด
3. งานค้างส่งงาน, Role Play ที่ยังไม่ส่งเตือนให้ทำสูง
4. หลักสูตรที่ผ่านคะแนน, เกรด, ทุน, ผ่านที่ดูย้อนหลังปานกลาง
5. AchievementBadges ที่ปลดล็อกสร้างแรงจูงใจต่ำ
🔄

ทำไมไม่ใส่ทุกอย่างในหน้าแรก?

❌ ผิด: ยัดทุกอย่างในหน้าแรก

ผู้ใช้จะหาอะไรไม่เจอ, เลื่อนหน้าจอยาก, โหลดช้า, สับสน — "Information Overload"

✅ ถูก: Progressive Disclosure

แสดงสิ่งที่สำคัญที่สุดก่อน (KPI + งานค้าง) → กดดูรายละเอียดได้ทีหลัง — หน้าแรกโหลดเร็ว ใช้งานง่าย

🔄 Learning Journey — วงจรการเรียนรู้

ออกแบบให้เห็น "เส้นทางพัฒนา" ชัดเจน ตั้งแต่ Day 1 ถึง ผู้นำ

🎯

หลักการออกแบบ Learning Journey

💡 ทำไมต้องแสดงเป็น Timeline?

สมาชิกต้องการเห็น "ฉันอยู่ไหน → ต้องไปไหน → อีกไกลไหม" — Timeline ตอบ 3 คำถามนี้ในหน้าเดียว ถ้าแสดงเป็นตาราง สมาชิกจะไม่เห็นภาพรวม

✅ ดีไซน์ที่เลือก

Vertical Timeline — อ่านจากบนลงล่าง (ธรรมชาติ) — แต่ละก้าวมีสีแยก 3 ทุน — กดแต่ละก้าว → ดูรายละเอียดหลักสูตร

📍

Flow Diagram — 9 ก้าวการพัฒนา

👋 สมาชิกใหม่ → 📚 ปฐมนิเทศ → 🌱 ทุนมนุษย์ → 💼 ทำงานจริง → ✅ ผ่านประเมิน → 🎓 ทุติยนิเทศ → 🤝 พี่เลี้ยง → 🏆 ตติย→จตุร→ปัญจ → 👑 ภาวะผู้นำ
💡 ทำไมแสดง 3 ทุนในแต่ละก้าว?

การพัฒนาไม่ใช่แค่ "อบรม" — ต้องพัฒนาทั้ง 3 ทุนพร้อมกัน: องค์กร (เข้าใจ PKG) + มนุษย์ (ทักษะ) + สารสนเทศ (เครื่องมือ) — ถ้าขาดทุนใดทุนหนึ่ง การพัฒนาไม่สมบูรณ์

📊

ทำไมไม่ใช้ Spiral/Circular Diagram?

✅ Vertical Timeline (เลือก)

• อ่านง่าย (บน→ล่าง)
• เห็นความคืบหน้า (เลื่อนลง)
• Mobile-Friendly
• เพิ่มก้าวได้ง่าย

❌ Spiral/Circular (ไม่เลือก)

• อ่านยาก (ต้องหมุนตา)
• ไม่เห็นลำดับชัดเจน
• Mobile ยากมาก
• เพิ่มก้าวต้องออกแบบใหม่

⚡ Workflow Design — ออกแบบขั้นตอนการทำงาน

ทุก Workflow ต้องเสร็จใน ≤ 3 คลิก

✅

Workflow 1: อนุมัติงาน (ADMIN)

1. เห็นงานค้างใน Dashboard → 2. กด "ดูรายละเอียด" → 3. กด "อนุมัติ" หรือ "ปฏิเสธ"
✅ เหตุผล

งานค้างแสดงใน Dashboard — ADMIN ไม่ต้องไปค้นหา → กดอนุมัติได้ทันที จาก 5-6 คลิก → เหลือ 3 คลิก

📝

Workflow 2: สมัครอบรม (สมาชิก)

1. เห็นหลักสูตรถัดไปใน Dashboard → 2. กด "สมัคร" → 3. ยืนยัน (เสร็จ!)
✅ เหตุผล

หลักสูตรถัดไปแสดงใน Dashboard — สมาชิกไม่ต้องค้นหา → สมัครได้ทันที ไม่ต้องกรอกฟอร์มยาว (ระบบรู้แล้วว่าใครสมัครอะไร)
